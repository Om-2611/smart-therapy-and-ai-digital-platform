## 1. AI Copilot — live in-call assistant (Stage 1)

Rewired the existing (NVIDIA-backed) live analysis onto our current stack. The copilot reads the same Firestore session inputs the report uses and runs on the therapist's "AI Copilot" button.

- **LLM → OpenRouter `gpt-oss-120b`.** Added `COPILOT_MODEL` + `chatJSON` + tolerant `extractJSON` to `src/lib/report/openrouter-client.ts`; `src/lib/rag/analysis.ts` now calls OpenRouter instead of NVIDIA NIM. New env var `OPENROUTER_COPILOT_MODEL` (defaults to `openai/gpt-oss-120b:free`). NVIDIA is retained **only** for embeddings.
  - Probe finding: forced `response_format: json_object` on gpt-oss mangles output (~35 s, leaked `<|constrain|>` tokens) — so we prompt for JSON and parse tolerantly instead (~12 s, clean).
- **Two-call safety pass** (`src/lib/rag/safety.ts`): runs in parallel with the suggestion; returns `none` / `watch` / `urgent` and is **authoritative** for the risk flag (general models were inconsistent at catching safety cues).
- **Module activity + in-call notes** now feed the copilot (`getSessionActivity` → `RAGContext` → prompt), so suggestions reflect what was *done*, not just said.
- **`copilotEvents[]` audit log** appended on every press (timestamp, summary, module, risk level) — survives for the end-of-session report.
- **Fixed a pre-existing 400 bug**: the session page sent `client` but `/api/ai-insight` expects `clientProfile`, so the button failed every time.

Model probe result (live, free models): `gpt-oss-120b` was the only one that reliably surfaced suicidal-ideation cues and stayed available; Qwen/Llama free were rate-limited (provider "Venice").

## 2. Bug — sessions not saved / not listed

**Cause:** the patient-card "Start Session" button linked to `/session/new?clientId=…`, but there is no `/session/new` route, so it fell through to `[sessionId]` with the literal id `"new"`. No Prisma `Session` row was ever created (the end PATCH 404'd), so the call never appeared in any list and repeated calls collided on the same Firestore `sessions/new` doc.

**Fix:** `src/app/clients/page.tsx` now `POST`s `/api/sessions` to create a real `SCHEDULED` row, then navigates to the real id (mirrors the home page's instant-session flow).

---

## 3. Bug — "join room" card never goes away

**Cause:** the home dashboard "Session Rooms" card listed **every** session (no status filter), so a `COMPLETED` session kept showing an "Enter Room" button — it looked like the session never ended. (DB confirmed the sessions were `COMPLETED` and had reports.)

**Fix:** `src/app/page.tsx` filters "Session Rooms" to joinable (`SCHEDULED` / `ACTIVE`) sessions only.

---

## 4. Bug — dropdowns unselectable across the project

**Root cause (confirmed in real Chrome via a Playwright probe):** Base UI's Dialog defaults to **`modal` mode**, which makes everything outside the dialog inert. A Select's options popup portals to `<body>` (outside the dialog), so its anchor positioner never ran — the popup collapsed to **0×0**. Options rendered as text but had no clickable area. The most-used dropdowns (Start Session / Schedule modals) were affected.

**Fixes:**
- `src/components/ui/dialog.tsx` — the `Dialog` wrapper now defaults to **`modal={false}`** (overridable). Verified: option size went 0×0 → 105×18 and selection works.
- `src/app/globals.css` — repointed the dead Radix z-index rule (`[data-radix-popper-content-wrapper]`) to the Base UI popup (`[data-slot="select-content"]`); also set the Select positioner z-index in `select.tsx`.
